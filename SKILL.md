## 📋 完整工作流程

### 每日6:30自动执行（定时Cron）— 5阶段10步

**第一阶段：48h未跟进复查（立即执行）**
```
① 查询多维表格，筛选【整改进度】=「待处理」且差评时间超48小时
② 发现超48h工单 → 立即生成【紧急督办】卡片 → 私信租户晁留柱
```

**第二阶段：24h内新差评采集**
```
③ miaoda搜索携程该酒店最新差评（限定24小时内）
④ miaoda搜索Trip.com该酒店最新英文差评（限定24小时内）
⑤ miaoda搜索去哪儿该酒店最新差评（限定24小时内）
⑥ miaoda搜索小红书该酒店相关投诉（限定24小时内）
```

**第三阶段：AI分析与归档**
```
⑦ AI自动12维度分类 + 严重度评分
⑧ 每条差评生成5Why根因 + 整改要求 + AI回复话术
⑨ 写入飞书多维表格【差评维度层】
```

**第四阶段：工单推送**
```
⑩ 按严重度分配工单：
    严重度≥8 → 立即私信负责人
    严重度7 → 工作时间内推送
    严重度<7 → 汇总后统一推送
```

**第五阶段：汇总汇报**
```
⑪ 生成每日差评清单卡片 → 私信汇报给决策者
    包含：类型分布/重点工单/48h复查结果/本周整改率
```

### 单次分析（手动触发）

用户说"分析XX酒店差评" → 执行①-⑪全流程

**48小时复查规则（必须每次执行）**：
```
每72小时复查一次所有「待处理」工单：
- 差评时间超48小时未跟进 → 私信租户督办
- 差评时间超48小时未回复 → 私信租户督办
```

## ✅ 美誉度闭环管理流程

```
差评发现
    ↓
生成工单（DR编号）
    ↓
24小时内分配给美誉度小组成员
    ↓
可自行回复 → 使用AI回复话术 → 粘贴至平台后台
    ↓
需跨部门协作 → 转交对应部门负责人
    ↓
跟进整改 → 回复差评 → 整改完成
    ↓
更新工单状态为「已整改」→ 闭环存档
```

**时效要求**：
- 严重度≥8：12小时内回复（紧急）
- 严重度≥7：24小时内回复（重要）
- 严重度<7：72小时内回复（常规）
- 整改完成：7个工作日内

## ⚠️ 定时任务超时处理（重要）

### 失败原因分析（2026-04-19早6:30）
```
错误：Request timed out before a response was generated
耗时：77242ms（约77秒）
原因：差评采集+AI分析+写入多维表格+发送消息，总步骤超过默认超时限制
```

### 解决方案

**方案1：增加cron任务超时配置（推荐）**
- 在cron payload中设置 `timeoutSeconds: 300`（5分钟）
- 每次执行保证足够时间完成全流程
- 晁总可通过OpenClaw配置界面或让AI助手执行

**方案2：分阶段执行（如果超时仍发生）**
```
Phase 1（定时触发）：
①-⑥ 只做数据采集+AI分析（不写入多维表格，不发消息）
⑦ 将分析结果写入临时文件或直接推送摘要

Phase 2（立即跟进）：
⑧-⑩ 写入多维表格+发送消息（作为单独的后续步骤）
```

**方案3：提高系统默认超时**
- 在OpenClaw配置中设置 `agents.defaults.timeoutSeconds: 300`

### 当前推荐配置
```yaml
cron_job_payload:
  timeoutSeconds: 300   # 必须设置为5分钟，避免超时
  # 不要依赖默认超时，要显式声明
```

## 🔄 48小时未跟进工单督办机制

### 规则
```
复查触发：每日8:00自动执行（差评清单发送后1.5小时）
检查范围：多维表格中【整改进度】为「待处理」且超过48小时的工单
处理方式：立即私信汇报给【租户晁留柱】（ou_fc1e75d64fec6e10ce94a51adc6f6409）
```

### 判定标准
```
「未跟进」定义满足任一：
- 工单创建后48小时，【整改进度】仍为「待处理」
- 工单创建后48小时，【是否已回复】仍为「否」
- 工单创建后48小时，无任何【整改措施】记录
```
